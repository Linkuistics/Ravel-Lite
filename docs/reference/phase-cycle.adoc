= Phase cycle

The phase cycle is the contract Ravel-Lite enforces against every plan it runs. Each cycle is the same fixed sequence — four reasoning phases interleaved with four audit-trail commit phases — and each phase has a precise contract for what it reads, what it writes, what it decides, and how it transitions. Once you understand the cycle, the rest of Ravel-Lite is bookkeeping around it.

This page documents each phase one by one, then covers the two cross-cutting mechanisms a reader needs to reason about a cycle holistically: the cross-plan findings inbox and the out-of-cycle truth-maintenance pass (planned `ravel-lite curate`, not yet shipped). Schemas for the files referenced live on the link:state-files.html[State files] and link:cross-plan-files.html[Cross-plan files] reference pages — this page links rather than re-narrates.

== The shape of a cycle

----
triage → git-commit-triage →
work → git-commit-work →
analyse-work → git-commit-analyse-work →
reflect → git-commit-reflect → [continue?] → triage
----

Every phase is a separate process invocation. Reasoning phases are agent processes (Claude Code or Pi, depending on the project's `agent:` setting) reading a prompt file under `defaults/phases/`. Audit-trail phases — the `git-commit-*` entries — are pure Rust handlers inside the orchestrator. They do no LLM work; they exist so each reasoning phase's mutations land in their own commit before the next one starts.

The cycle is driven by `<plan>/phase.md`, a one-line file naming the phase to run next. The orchestrator reads it at the top of each iteration and dispatches to the matching handler. Every phase, before it stops, rewrites this file to name its successor — for reasoning phases via `ravel-lite state set-phase`, for audit-trail phases via a direct write. A failure to advance is a stop condition: the loop refuses to re-invoke the same phase twice in a row, which would otherwise mask infinite-loop bugs.

== Reasoning phases

The four reasoning phases each spawn the configured agent against a prompt file in `defaults/phases/`. They are documented below in cycle order — Triage opens, Reflect closes. Prompts are templated — `{{PLAN}}`, `{{PROJECT}}`, `{{ORCHESTRATOR}}`, and similar tokens are substituted before invocation. Customisation goes through `ravel.append_prompt` in `config.lua`; see the link:config-and-prompts.html[Configuration and prompts] reference.

=== Triage

The cycle-opening phase. Reviews the backlog, mines completed tasks for hand-offs, reprioritises, and writes findings to the context-level inbox if any cross-plan observation surfaced.

Reads:: The backlog and current memory. Triage explicitly does *not* read sibling, parent, or child plan content — cross-plan awareness comes from the `Related plans` block in its prompt (paths only), never from direct foreign reads. No subagent is dispatched against a peer plan.

Writes:: Backlog mutations (`set-status`, `set-title`, `reorder`, `add`, `delete`, `set-dependencies`, `clear-handoff`); occasionally memory adds when archiving a hand-off. Cross-plan observations are written to the context-level findings inbox via `ravel-lite findings add` — a user-mediated queue the operator processes out of band.

Decisions:: Which tasks are still relevant; what to reprioritise and why; whether each completed task's hand-off warrants a new top-level task or a memory archive; whether any observation belongs in the findings inbox.

Transition:: `ravel-lite state set-phase {{PLAN}} git-commit-triage`.

Triage reprioritisations require a stated reason naming the cycle learning that drove the change — "felt better here" is not a reason. The narrative preamble carries `[REPRIORITISED]`, `[PROMOTED]`, `[ARCHIVED]`, `[BLOCKER]`, and `[FINDING]` intent labels alongside the renderer's structural diff. The link:cross-plan-files.html[Cross-plan files] reference covers `findings.yaml` in detail.

=== Work

The only interactive phase. The user steers task selection at the prompt; the agent then implements the chosen task and records results.

Reads:: `<project>/README.md`, the backlog (`ravel-lite state backlog list`), distilled memory (`ravel-lite state memory list`), and declared peer-project relationships (`ravel-lite state related-components list`). Per-language coding-style files under `<orchestrator>/fixed-memory/` are read on demand, only when about to touch code.

Writes:: Whatever the chosen task requires — source files, tests, docs, configuration — plus the task's status flip and `Results:` block in `backlog.yaml`. Source-file changes are deliberately *not* committed by the work phase; they are left dirty so analyse-work can author the commit message from the actual diff.

Decisions:: Which task to run; how to scope it; when to declare it done.

Transition:: `ravel-lite state set-phase {{PLAN}} git-commit-work`.

The work phase is the only point in the cycle where the user is in the loop. Every later phase is headless — once you confirm "Proceed to next work phase?" the cycle runs unattended through to the next work prompt.

=== Analyse-work

Examines the actual git diff produced by the work phase, writes a session-log entry, and authors `commits.yaml` — an ordered specification of the commits the next phase should apply.

The phase explicitly does not trust self-reports. Its prompt feeds it two pre-computed inputs: a `{{WORK_TREE_STATUS}}` snapshot of dirty paths captured the moment work exited, and a `{{BACKLOG_TRANSITIONS}}` block diffing `backlog.yaml` against the work baseline. Both are computed by the orchestrator in code rather than asked of the LLM, on the principle that mechanical work belongs in Rust.

Reads:: The work-baseline SHA from `<plan>/work-baseline`; the diff against that baseline; the backlog (for results context). Memory and peer-project relationships are explicitly skipped — they are not relevant to summarisation.

Writes:: The session record (`ravel-lite state session-log set-latest`), `commits.yaml`, and any task-status repairs (`ravel-lite state backlog repair-stale-statuses`) for tasks whose results were recorded but status not flipped. The phase may also revert accidental edits left in the work tree.

Decisions:: How to partition the diff into commits (default: one commit per cycle covering the whole subtree); whether to promote a hand-off to a new backlog task or archive it; whether to revert any accidental edits; commit titles and bodies.

Transition:: `ravel-lite state set-phase {{PLAN}} git-commit-analyse-work`.

The `commits.yaml` shape is a list of `{paths: […], message: <title + body>}` entries. Pathspecs use git's standard syntax — globs (`src/**`) and exclusions (`:!src/generated/`) both work. If the file is missing or malformed, `git-commit-analyse-work` falls back to a single catch-all commit; that fallback is a safety net, not the intended path.

=== Reflect

Distils the latest session's learnings into durable memory. Reflect is the only lossy phase — it adds, sharpens, contradicts, and prunes memory entries based on what the session uncovered.

Reads:: The latest session record (`ravel-lite state session-log show-latest`), current memory (`ravel-lite state memory list`), and the universal Memory style rules at `<orchestrator>/fixed-memory/memory-style.md`.

Writes:: New, updated, or deleted memory entries via `ravel-lite state memory add` / `set-body` / `set-title` / `delete`.

Decisions:: For each learning — is it new, sharpening an existing entry, contradicting one, or making one redundant? Pruning is aggressive: memory should hold only what is currently true and useful, with the session log as the safety net for discarded content.

Transition:: `ravel-lite state set-phase {{PLAN}} git-commit-reflect`. Reflect *always* names `git-commit-reflect`, and `git-commit-reflect` closes the cycle.

Reflect deliberately does not read the backlog or the session log history. The first avoids task-oriented thinking during reflection; the second keeps the audit trail strictly append-only and never an LLM input.

Truth maintenance — consolidating overlapping memory entries and tightening verbose prose — is now an *out-of-cycle* operation. The planned `ravel-lite curate` command (not yet shipped) is the user-invoked compaction pass; it has no role inside the in-cycle phase loop. The earlier `dream` phase, which ran conditionally inside the cycle, has been removed entirely.

== Audit-trail phases

The `git-commit-*` phases are Rust handlers, not agent invocations. Each commits the prior reasoning phase's mutations and prepares the baseline file the next reasoning phase will diff against. They exist so every state transition lands in its own commit, producing the per-cycle audit trail visible in `git log`. The four handlers are documented below in cycle order.

=== git-commit-triage

Opens the cycle. Commits the triage phase's plan-state mutations, captures `work-baseline`, and writes `work` to `phase.md` so the cycle proceeds to the user-interactive work phase.

The order of operations is load-bearing. The triage commit lands first, advancing HEAD; *then* `work-baseline` is captured, so analyse-work later in this cycle diffs against post-triage state rather than conflating triage's edits into `{{BACKLOG_TRANSITIONS}}`. The baseline lands as a separate `save-work-baseline` commit so the working tree stays clean at the user prompt that follows.

Transition:: `phase.md` ← `work`. The handler returns `true` to the inner phase loop, which continues into the work phase.

=== git-commit-work

Sits between work and analyse-work. Commits any plan-state changes the work agent made in-flight (memory entries, `focus-objections.yaml`, in-line backlog status flips). Source-tree changes from work remain *deliberately dirty* for analyse-work to inspect via the work-tree snapshot — they are not committed here, only at git-commit-analyse-work after `commits.yaml` has been authored.

After the plan-state commit lands, the handler captures `analyse-work-baseline` (the post-commit HEAD) so analyse-work has a stable diff anchor, then writes the baseline as a separate `save-analyse-work-baseline` commit.

Transition:: `phase.md` ← `analyse-work`.

=== git-commit-analyse-work

Sits between analyse-work and reflect. Applies `commits.yaml` to the project subtree, then captures `reflect-baseline`. This is the only audit-trail phase that may produce more than one commit.

It reads `commits.yaml` and applies each entry in order — `git reset HEAD -- .`, `git add -- <paths>`, `git commit -m <message>` — producing one commit per entry with content. If `commits.yaml` is missing or malformed, a single catch-all commit covers the whole subtree under `run-plan: analyse-work ({{PLAN}})`.

Once the work-commits land, the handler appends `latest-session.yaml` to `session-log.yaml` (mechanical mirror write — kept in code so a phase prompt cannot forget it), commits any plan-state edits analyse-work itself produced (`latest-session.yaml`, backlog status repairs), and writes `reflect-baseline` capturing the post-commit HEAD. A follow-on `save-reflect-baseline` commit lands the baseline file. Then it warns if the project tree is still dirty: anything left uncommitted means the work agent edited paths that no `commits.yaml` entry covered, which is the silent failure mode the design exists to surface.

Transition:: `phase.md` ← `reflect`.

=== git-commit-reflect

Closes the cycle. Commits reflect's memory mutations, captures `triage-baseline`, writes `triage` to `phase.md` so the next cycle resumes at triage, and exits the inner phase loop.

Two commits land: `run-plan: reflect ({{PLAN}})` for the memory mutations, then `run-plan: save-triage-baseline ({{PLAN}})` for the baseline file. The phase marker is written to `phase.md` *before* the commits so a fresh `ravel-lite run` after a crash mid-cycle resumes at triage rather than re-running reflect.

Note that `latest-session.yaml` is intentionally not touched here — analyse-work overwrites it next cycle, and leaving it in place keeps the prior session's record available for operator inspection in the gap between cycles.

Transition:: `phase.md` ← `triage`. Then the handler returns `false` to the inner phase loop, which exits. The outer driver decides whether to run another cycle (single-plan mode prompts "Proceed to next work phase?"; multi-plan mode returns to its survey-based selection loop).

== Out-of-cycle truth maintenance

Memory compaction — consolidating overlapping entries and tightening verbose prose — does not run inside the phase cycle. The planned `ravel-lite curate` command is the user-invoked, context-wide truth-maintenance pass over every active plan's memory and every component memory in every registered repo. As of this writing the verb is not yet shipped.

The earlier in-cycle `dream` phase (which fired conditionally based on a memory word-count headroom) has been removed entirely. Plan execution never blocks on truth maintenance; the user invokes curate on demand when memory accumulation warrants it.

== Cross-plan findings

When reflect or triage surfaces an observation that is out of scope for the current plan but worth attention elsewhere — a defect noticed in code that was read but not edited, a suggestion that fits a different plan, an Atlas component issue picked up in passing — the phase records it in the *context-level* findings inbox via `ravel-lite findings add`. Findings are advisory: no driver step auto-acts on them, and no phase consumes them. The user processes the inbox out of band.

Reflect and triage both call into the same verb:

[source,bash]
----
ravel-lite findings add \
    --claim "<one-line assertion>" \
    --body-file <path> \
    --raised-in {{PLAN}} \
    --authored-in <reflect|triage> \
    [--component <repo>:<component-id>]
----

Findings live at the context level alongside `config.lua` and `repos.yaml` — *not* inside any plan directory — so observations raised in one plan are visible to any other cycle inspecting the inbox. The link:cross-plan-files.html[Cross-plan files] reference documents the schema and status transitions in full.

Both phases skip the step entirely when no out-of-scope observation surfaced; that is the default case. The narrative preamble carries one `[FINDING] <one-line claim>` line per finding written, so the structural diff and the operator's TUI both reflect the writes.

The previous mechanism — `subagent-dispatch.yaml`, a per-plan scratch file the driver consumed by spawning parallel subagents against named peer plans — was retired. Automated cross-plan dispatch was prone to producing low-quality drive-by edits in target plans without the originating cycle's full context; the user-mediated inbox keeps the human in the loop without losing the original observation.

Targets must be absolute paths. The orchestrator looks up the agent backend's `dispatch_subagent` implementation; for Claude Code that means a fresh `claude --add-dir <target>` subprocess scoped to the target plan. The dispatching plan's only role is to write the YAML; everything else — agent spawning, parallelism, log routing, file cleanup — is the driver's responsibility. See the link:cross-plan-files.html[Cross-plan files] reference for the full schema and dispatch lifecycle.

== Baselines

Four `<phase>-baseline` files thread the cycle. Each captures the HEAD SHA at a specific point so the next reasoning phase can diff against it:

work-baseline:: Captured by `git-commit-triage`. The diff anchor analyse-work uses to compute `{{BACKLOG_TRANSITIONS}}` and `{{WORK_TREE_STATUS}}`. On a brand-new plan with no prior cycle, it is seeded from the first work phase's pre-invocation HEAD so analyse-work always has something to diff against.

analyse-work-baseline:: Captured by `git-commit-work`. Anchors the post-work-commit HEAD so analyse-work and downstream tooling can locate the boundary between work's plan-state edits and the source-tree changes still pending in `commits.yaml`.

reflect-baseline:: Captured by `git-commit-analyse-work`. Read by reflect's `state phase-summary render --baseline` invocation to produce its structural memory diff.

triage-baseline:: Captured by `git-commit-reflect`. Read by triage's `phase-summary render --baseline` at the start of the next cycle.

Each baseline lands in its own commit (`save-<phase>-baseline`) rather than being amended onto the prior commit. The follow-on-commit pattern keeps the working tree clean at the user prompt and avoids orphaning the SHA the file just captured.

== Recovery and inspection

Every state transition is a file write or a commit. Two consequences worth knowing:

A crash mid-cycle is recoverable from `phase.md`. Whichever phase was running when the orchestrator died is the phase that runs again on next invocation; phase prompts are designed to be idempotent against partial prior runs (e.g., reflect's `state memory add` against an entry that already exists will fail loudly rather than duplicate).

A complete cycle leaves a complete history. `git log --oneline` between two `save-triage-baseline` commits is the full record of one cycle: the triage commit, the work commit(s), the reflect commit, and the `save-*-baseline` commits between them. A reader can scrub through a cycle the same way they would scrub through a video.

Phase mutations made by hand-edit between cycles are first-class inputs. Editing `backlog.yaml` between work and analyse-work is just another input the next phase will read; editing `memory.yaml` between cycles is fine. The "no magic" principle means there is no shadow state — what you see in the files is what the next phase sees.
