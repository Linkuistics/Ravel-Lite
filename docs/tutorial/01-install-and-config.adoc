= Install and configure

Ravel-Lite is a single Rust binary plus an optional config directory. The binary lives on `PATH`; the config directory holds your customisations — there is no required on-disk state, because the shipped prompts and agent definitions are embedded in the binary itself. This chapter installs the binary, scaffolds the config directory, and points your shell at it.

The walk-through assumes macOS or Linux; the steps are identical on either. Where a path differs the difference is called out.

== Install via Homebrew

The simplest install on either platform is the bottled Homebrew formula:

[source,bash]
----
brew install linkuistics/taps/ravel-lite
----

Updates flow through the same channel:

[source,bash]
----
brew upgrade ravel-lite
----

Homebrew installs a precompiled binary, so the user does not need a Rust toolchain. The four target tarballs (Intel and Apple-Silicon macOS, x86_64 and aarch64 Linux glibc) are built and published from a developer machine via `cargo-release` plus the two scripts under `scripts/release-*.sh` — the published Linux binaries pin glibc to 2.17 for wide compatibility.

Verify the install:

[source,console]
----
$ ravel-lite version
ravel-lite 1.0.3 (v1.0.3, built 2026-04-27T12:13:39Z)
----

The three components are the crate version, `git describe --tags --always --dirty` from the build, and the UTC build timestamp. A running binary therefore always identifies the commit it was built from — useful when reporting an issue.

[NOTE]
====
If you do not use Homebrew, clone the repository and `cargo install --path .` against a stable Rust toolchain. The release tarballs pinned to glibc 2.17 are intended for users who want a single binary; building from source is the supported alternative for everyone else.
====

== `ravel-lite init`

`init` scaffolds a *ravel-context* — the user's single root holding configuration, the repo registry, and every plan. It does *not* materialise the shipped prompts, agent definitions, or fixed-memory files — those are baked into the binary and read from the embedded set on every invocation. The user-owned files `init` seeds are: a commented-out `config.lua` starter, an empty `repos.yaml`, an empty `findings.yaml`, and the four override-point subdirs (`agents/`, `phases/`, `fixed-memory/`, `plans/`). It also gives the context its own git identity (`git init` plus an initial empty commit) so plan-state changes can be committed to the context's history independently of any target repo.

`init` is path-optional. With no `--config` flag it resolves the same precedence chain as every other subcommand (`$RAVEL_LITE_CONFIG`, then the platform default at `~/.config/ravel-lite/`). The example below names the path explicitly to keep the macOS and Linux walkthroughs identical:

[source,console]
----
$ ravel-lite init --config ~/.config/ravel-lite
  ✓ Scaffolded ravel-context at /Users/admin/.config/ravel-lite
    - config.lua (commented-out starter; embedded defaults in effect)
    - repos.yaml (empty repo registry)
    - findings.yaml (empty findings inbox)
    - agents/, phases/, fixed-memory/, plans/ (empty subdirs)
    - .git (context owns its own history)

  Config is at the default location; ravel-lite will discover it automatically.
----

If you target a non-default path, the trailing guidance instead instructs you to set `RAVEL_LITE_CONFIG` or pass `--config` on each invocation.

After `init` runs, the directory holds the v2 layout:

[source,console]
----
$ ls ~/.config/ravel-lite
agents  config.lua  findings.yaml  fixed-memory  phases  plans  repos.yaml
----

The four subdirs (`agents/`, `phases/`, `fixed-memory/`) are *override points* — they are empty by default, and the binary serves the embedded set unless you drop a same-named override here. `plans/` is where every plan you create lives; `repos.yaml` is the registry of repos you can target with a plan; `findings.yaml` is the cross-cutting inbox written by triage and reflect.

That file is the customisation surface. It is plain Lua and entirely commented out at scaffold time, so reading the file is the fastest way to discover what you can customise:

[source,lua]
----
-- ravel-lite config (Lua)
--
-- This file is optional. Out of the box, ravel-lite reads every
-- shipped prompt and agent definition from its embedded set, so an
-- empty config dir works out of the box.
--
-- Customise behaviour by uncommenting and editing the calls below.
-- See `ravel-lite reference config-and-prompts` for the full API.
--
-- ravel.set_agent("claude-code")
-- ravel.set_model("work", "claude-opus-4-7")
-- ravel.append_prompt("work", [[
--   Project-specific guidance appended after the embedded work prompt.
-- ]])
----

The link:../docs/config-and-prompts.html[Configuration and prompts] reference covers the full Lua API: setters for the active agent and per-phase models, an `append_prompt` registrator that adds extra guidance to the embedded prompts, and direct writes to `ravel.config` for headroom and other scalars.

== Re-running `init`

Re-running `init` against an existing context is safe. It does not error, it does not overwrite your `config.lua` / `repos.yaml` / `findings.yaml`, and it does not re-run `git init`:

[source,console]
----
$ ravel-lite init --config ~/.config/ravel-lite
  ✓ Context dir already initialised at /Users/admin/.config/ravel-lite

  Config is at the default location; ravel-lite will discover it automatically.
----

`--force` exists for a narrow purpose: pruning paths that *used to ship* and have since been removed or renamed. It does not refresh shipped defaults — they live in the embedded set, not on disk — and it deliberately leaves your user-owned files (`config.lua`, `repos.yaml`, `findings.yaml`) untouched.

[source,console]
----
$ ravel-lite init --config ~/.config/ravel-lite --force
  ✓ Init --force complete: 0 pruned (shipped defaults are embedded; nothing to refresh on disk)

  Config is at the default location; ravel-lite will discover it automatically.
----

In day-to-day use you almost never need `--force`. The one situation that warrants it is upgrading across a release that retires a previously-shipped path: `--force` removes the now-defunct file or directory so your config tree matches the new layout. The `--help` text and release notes call out when this matters.

== Telling ravel-lite where the config directory is

If you used the platform default location, you can skip this section. Otherwise, ravel-lite needs to know where to look. The precedence chain is:

. `--config <path>` on the command line, if supplied;
. `RAVEL_LITE_CONFIG` environment variable, if set;
. the platform default (`~/.config/ravel-lite/` on Linux, `~/Library/Application Support/ravel-lite/` on macOS).

If none of the three resolves to an existing directory, ravel-lite still runs against the embedded defaults — but customisations in your `config.lua` will not be picked up, because nothing tells the binary where to find it. The recommended workstation setup is to set the environment variable in your shell profile:

[source,bash]
----
# ~/.zshrc or ~/.bashrc
export RAVEL_LITE_CONFIG="$HOME/.config/ravel-lite"
----

Reload the shell or run `source ~/.zshrc`, then verify:

[source,console]
----
$ ravel-lite state projects list
schema_version: 1
projects: []
----

An empty catalog is valid output — it just means no projects have been registered yet. We will add one in the next chapter.

The `--config` flag exists for one-off overrides: running an isolated config directory for an experiment, or driving multiple installations from the same shell. It overrides the env var without changing it.

== Sanity check

Two signals tell you the install is healthy.

First, `ravel-lite version` returns a version line — that confirms the binary is on `PATH` and runs. Second, `ravel-lite state projects list` returns the empty catalog (or any catalog you have already populated):

[source,console]
----
$ ravel-lite state projects list
schema_version: 1
projects: []
----

That confirms the binary can find a config directory and parse it. There is no third check for shipped prompt files or an `agents/` directory — those are embedded in the binary, not on disk, so a missing `phases/` or `agents/` directory in your config dir is not a fault. If you want to inspect the shipped prompts, the source-of-truth is `defaults/` in the Ravel-Lite source tree; the link:../docs/config-and-prompts.html[Configuration and prompts] reference documents how to override or append to them via Lua.

With the binary installed and the config directory pointed at, you are ready to set up a project — the link:02-the-project.html[next chapter] introduces the worked-example reading-list-manager that the rest of the tutorial threads through.
